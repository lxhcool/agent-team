# Skill 包分析器 MVP 设计

日期：2026-05-07  
状态：待评审

## 1. 背景

当前团队对 skill 的使用方式高度依赖口碑传播：

- 听说某个 skill 好用就直接拿来用
- 不清楚它真正解决什么问题
- 不清楚它为什么有效
- 不清楚它依赖哪些前提
- 不清楚它是不是只是“看起来好用”

把单个 `SKILL.md` 或 prompt 文件直接丢给通用大模型分析，通常只能得到一份“解释型总结”，例如：

- 这个 skill 覆盖面很全
- 它的流程很系统
- 它有分层结构
- 它工程化做得不错

这些结论不一定错，但它们无法支撑团队判断。团队真正需要的是：

- 这个 skill 包到底由什么组成
- 它实际怎么运作
- 它为什么不是一个普通提示词
- 哪些部分真的决定效果
- 哪些部分非常脆弱

因此，第一版不做“skill 平台”，也不做“通用 AI 资产治理系统”。第一版只做一个最小而明确的工具：

**导入一个本地 skill 路径，分析这个 skill 包为什么有效。**

## 2. 产品定义

这是一个网页工具，不做桌面端。

用户输入一个本地路径：

- 可以是单个 `SKILL.md`
- 也可以是一个完整 skill 目录

系统读取该路径下的 skill 包内容，并输出一页结构化分析卡片，帮助普通团队成员理解这个 skill 是干什么的，以及它为什么有效。

## 3. 目标用户

第一版只服务一种主要用户：

**普通团队成员**

他们通常不是 skill 的作者，也不是平台管理员。他们最先想知道的是：

- 这个 skill 是干什么的
- 为什么它值得用

他们不关心过深的工程细节，也不需要一开始就管理整个平台。

## 4. 核心目标

第一版只验证一件事：

**我们能否对一个完整 skill 包给出比通用大模型“读后感”更有判断力的机制拆解结果。**

用户在看完分析结果后，至少应该能回答：

1. 这个 skill 主要解决什么问题
2. 它实际是怎么跑起来的
3. 它为什么比普通写法更强
4. 哪些部分是效果杠杆
5. 哪些地方容易失效

## 5. 非目标

第一版明确不做：

- skill 平台
- 团队共享
- 资产库
- 多 skill 对比
- 自动组合 / 缝合推荐
- 改进建议
- 团队治理仪表盘
- 桌面 App 支持

这些都属于后续问题。第一版不能被平台化诉求拖死。

## 6. 输入边界

第一版输入对象为：

**一个本地路径**

支持两种情况：

1. 单个 `SKILL.md` 文件
2. 一个完整的 skill 目录

如果是目录，系统需要一起分析：

- `SKILL.md`
- 被 skill 明确引用的脚本
- 模板文件
- 参考文件
- 局部资源目录

原则是：**分析对象是一个完整 skill 包，而不是只读说明文件。**

## 7. 核心输出

第一版输出是：

**完整的结构化分析卡片**

不是摘要页，不是流程图主视图，不是单段报告。

页面一打开就直接展示完整分析结果。  
顶部会有轻量目录/锚点，但主视图仍然是一页完整内容。

## 8. 分析卡片结构

第一版固定只有 5 个模块：

1. **它是干什么的**
2. **执行路径**
3. **为什么有效**
4. **关键杠杆**
5. **脆弱点**

其中：

- `为什么有效` 是默认高亮模块
- `它是干什么的` 用途导向，帮助普通成员先建立理解

### 8.1 它是干什么的

回答：

- 它解决什么问题
- 适合什么场景
- 主要给谁用
- 什么时候该用

### 8.2 执行路径

回答：

- 用户触发后从哪里开始
- 读取了哪些关键文件
- 依赖了哪些脚本或资源
- 实际执行顺序是什么

### 8.3 为什么有效

这是第一版最重要的模块，必须回答：

- 它为什么不是普通 prompt
- 它的效果来源到底是什么
- 是知识库、脚本、流程约束、分层设计还是外部依赖在起作用

### 8.4 关键杠杆

回答：

- 哪些部分决定了大部分效果
- 哪些部分是关键组成
- 哪些部分可以替换但影响较小

### 8.5 脆弱点

回答：

- 它强依赖哪些前提
- 哪些部分容易失效
- 哪些地方换模型后会明显变差

## 9. 输出数据结构

第一版建议落成固定结构，避免再次变成泛化报告：

```ts
type SkillPackageAnalysis = {
  summary: {
    title: string;
    packageType: "single_file" | "skill_package";
    purpose: string;
    effectivenessCore: string;
  };
  purpose: {
    whatItDoes: string;
    scenarios: string[];
    audience: string[];
    whenToUse: string[];
  };
  executionPath: {
    entry: string;
    steps: string[];
    dependencies: string[];
    touchedFiles: string[];
  };
  effectiveness: {
    primaryReasons: string[];
    supportingMechanisms: string[];
    notJustPromptBecause: string[];
  };
  leveragePoints: {
    criticalParts: string[];
    replaceableParts: string[];
    effectDrivers: string[];
  };
  fragility: {
    assumptions: string[];
    failureRisks: string[];
    modelSensitivity: string[];
  };
};
```

重点不是字段多，而是字段在逼系统做“机制判断”，而不是泛泛总结。

## 10. 页面结构

第一版页面采用：

- 单页纵向浏览
- 页内锚点导航
- 默认高亮 `为什么有效`

不采用传统左右后台结构。

页面从上到下分为：

1. 顶部导入区
2. 结果摘要区
3. 页内锚点导航
4. 五张结构化分析卡片

## 11. 页面交互

### 11.1 顶部导入区

包含：

- 本地路径输入框
- `开始分析`
- `重新分析`

第一版只需要支持手动输入路径，不需要拖拽上传，不需要文件选择器集成。

### 11.2 结果摘要区

展示：

- skill 名称
- 分析对象类型（单文件 / skill 包）
- 一句话用途总结
- 一句话“为什么有效”

### 11.3 锚点导航

锚点不是纯目录，而要带轻量判断信息。

每个锚点至少包含：

- 模块名称
- 一句短摘要或短标签

默认高亮：

- `为什么有效`

### 11.4 重新分析

第一版必须支持对同一路径重新分析。

这是必要功能，因为：

- skill 内容可能已经变化
- 用户可能想刷新结果
- 没有重新分析，页面会退化成一次性 demo

## 12. UI 风格要求

第一版 UI 风格必须：

**保留当前项目现有这套风格。**

具体要求：

- 延续当前白底、浅灰边框、柔和阴影、紫蓝强调色的视觉体系
- 不改成后台控制台风格
- 不改成强数据平台风格
- 不重新发明一套新的品牌语言

界面应当看起来像：

- 当前项目的同一产品族页面
- 面向普通成员的分析工具
- 信息清楚、分区明确、不会太技术化

## 13. 为什么这版有机会成立

第一版的价值不在于“总结一个 skill”，而在于它比通用大模型更进一步：

- 通用大模型擅长解释
- 这版 MVP 必须擅长机制拆解

也就是说，它不能停留在：

- 覆盖面很全
- 工程化很好
- 结构很清楚

而必须进一步说明：

- 真正起作用的是哪几层机制
- 为什么这些机制比普通提示词更有效
- 这个 skill 包依赖哪些关键结构才能稳定工作

如果做不到这点，这个工具没有必要存在。

## 14. 风险

第一版最容易失败的地方有四个：

1. 输出又变成“读后感”
2. 只分析 `SKILL.md`，没有一起分析脚本和依赖
3. 把页面做成技术控制台，普通成员看不懂
4. “为什么有效”最后还是空话，例如“因为结构完整、覆盖面大”

只要出现这些情况，MVP 就不成立。

## 15. 验收标准

第一版完成后，至少应满足：

1. 用户可以输入一个本地路径并成功开始分析
2. 系统能识别单文件和完整 skill 包两种对象
3. 页面能输出完整 5 模块结构化分析卡片
4. `为什么有效` 模块能指出明确机制，不是空泛形容
5. 页面支持重新分析同一路径
6. UI 风格与当前项目保持一致

## 16. 后续但不进入本期

本期结束后，如果 MVP 被证明有价值，再考虑：

- 多 skill 对比
- 组合 / 缝合判断
- 团队资产库
- 平台化治理能力
- 资产保留 / 淘汰建议

但这些都必须建立在第一版分析器本身成立的前提上。
