# Workspace Stage Model Design

日期：2026-05-08

## 1. 背景

当前 workspace 流程虽然已经具备阶段推进能力，但阶段命名和阶段职责对团队成员来说不够直观。

现有问题主要有：

1. 阶段名偏内部语义，不够口语化，团队成员一眼不容易理解。
2. 个别阶段之间存在职责重叠，尤其是前期确认与范围收束部分。
3. 阶段顺序和阶段产出之间不完全一致，容易让用户误解“当前到底要确认什么”。
4. 流程目标不是写代码，而是尽可能在前期把关键决策补齐并锁住，减少后续返工，但当前阶段定义没有完全围绕这个目标来设计。

因此需要重构阶段模型，让它更适合团队共用，也更贴近真实交付过程。

## 2. 设计目标

这套阶段模型需要满足以下目标：

1. 阶段标题必须口语化、业务化、陈述式，让产品、开发、运营、管理者都能快速理解。
2. 阶段设计必须围绕“逐步锁定关键决策”展开，而不是按岗位或内部过程拆分。
3. 每个阶段都必须有明确的输入、输出、确认动作和边界。
4. 主流程不默认按 MVP 组织，而是默认输出当前要做的完整方案。
5. 最终结果不是代码产物，而是可交接、可下载、可打包的文档集合。

## 3. 总体阶段结构

新的主流程固定为 5 个阶段：

1. 需求确认
2. 方案设计
3. 细节确认
4. 开发方案
5. 交付清单

这 5 个阶段的含义如下：

- `需求确认`：确认这次到底要做什么。
- `方案设计`：形成完整产品方案。
- `细节确认`：锁住容易引起返工的规则和边界。
- `开发方案`：形成开发可接手的实现方案。
- `交付清单`：整理全部已确认产物并提供下载。

## 4. 阶段定义

### 4.1 需求确认

#### 目标

把本次需求本身确认清楚，不只确认背景和目标，还要确认本次真正要做的核心内容。

#### 输入

- 用户原始描述
- 阶段对话补充
- 相关背景资料

#### 输出

一份 `需求确认文档`。

#### 文档内容

建议至少包含以下部分：

- 背景
- 目标
- 目标用户
- 当前约束
- 本次要做的核心内容
- 待确认问题

#### 确认动作

用户确认：这次要做的事情已经表达清楚，可以进入方案设计。

#### 边界

- 这个阶段不展开具体产品方案。
- 不进入模块设计、页面结构、交互细节、技术实现。
- 如果存在明显缺失信息，应优先暴露缺口，而不是假装已经确认完整。

### 4.2 方案设计

#### 目标

形成完整的产品方案，明确这个产品由哪些功能模块组成，这些模块之间如何配合，再落到页面结构和主要流程。

#### 输入

- 需求确认文档

#### 输出

一份 `方案设计文档`。

#### 文档内容顺序

方案设计阶段采用固定展开顺序：

1. 功能模块
2. 模块关系
3. 页面结构
4. 主要流程

#### 文档内容

建议至少包含以下部分：

- 产品整体方案说明
- 核心功能模块清单
- 模块职责说明
- 模块之间的关系
- 页面结构与页面关系
- 主要用户流程
- 后续可扩展方向（可选）

#### 确认动作

用户确认：整体方案方向成立，可以进入细节确认。

#### 边界

- 这个阶段默认覆盖本次要做的完整方案，不按默认 MVP 切分。
- 可以补充后续可能扩展的功能，但这些内容不能混入当前主交付范围。
- 不在这个阶段深入权限、状态流转、异常处理、数据口径等细节规则。

### 4.3 细节确认

#### 目标

锁住最容易引发返工的业务规则和边界细节，让后续开发方案有稳定依据。

#### 输入

- 方案设计文档

#### 输出

一份 `细节确认文档`。

#### 文档重点

这个阶段不以页面视觉或交互细节为主，而是优先确认业务规则和边界。

#### 文档内容

建议至少包含以下部分：

- 角色与权限边界
- 状态流转
- 异常处理
- 数据口径
- 特殊业务规则
- 重要边界条件
- 未决问题与影响说明

#### 确认动作

用户确认：关键规则和边界已经足够明确，可以支撑开发方案设计。

#### 边界

- 这个阶段不是技术设计阶段。
- 不负责给出技术架构、接口实现方案、依赖清单。
- 也不以页面美化、视觉表现或交互动效为主。

### 4.4 开发方案

#### 目标

在前面三个阶段的基础上，形成一份开发可直接接手的实现方案。

#### 输入

- 需求确认文档
- 方案设计文档
- 细节确认文档

#### 输出

一份 `开发方案文档`。

#### 文档内容

建议至少包含以下部分：

- 技术落地方式
- 模块拆分建议
- 数据与接口组织方式
- 外部依赖
- 风险与限制
- 开发实施建议
- 建议的先后顺序

#### 确认动作

用户确认：开发已经可以按这份方案开始实施。

#### 边界

- 这个阶段不写代码。
- 这个阶段不伪装成“已经开发完成”。
- 输出必须是方案、结构、约束和建议，而不是实现结果。

### 4.5 交付清单

#### 目标

把前面所有已确认文档整理成可交接、可下载的交付集合。

#### 输入

- 各阶段确认文档

#### 输出

一个 `交付清单`。

#### 交付形式

交付清单至少需要支持：

- 浏览所有已确认文档
- 单独下载某一份文档
- 打包下载全部文档

#### 内容结构

交付清单应至少包含：

- 文档名称
- 所属阶段
- 简要说明
- 下载入口

#### 确认动作

用户确认：本次可交付内容已经齐全，可供后续协作方直接取用。

#### 边界

- 这个阶段负责整理和交接，不再新增核心方案内容。
- 如果前置文档不完整，应回到对应阶段补齐，而不是在交付清单阶段临时兜底。

## 5. 阶段之间的关系

### 5.1 主线关系

阶段之间是逐步收紧的关系：

- `需求确认` 决定做什么
- `方案设计` 决定整体怎么设计
- `细节确认` 决定关键规则怎么定
- `开发方案` 决定怎么落地
- `交付清单` 决定如何交接

### 5.2 前后依赖

1. 没有明确需求确认，不进入方案设计。
2. 没有完整方案设计，不进入细节确认。
3. 没有关键规则确认，不进入开发方案。
4. 没有稳定开发方案，不进入交付清单。

## 6. 需要替换或删除的旧思路

### 6.1 删除单独的“范围与优先级”阶段

原因：

- 对当前目标来说，这个阶段容易和需求确认重复。
- 主方案默认就是本次要做的完整方案，不默认以 MVP 切分。
- 只有在用户明确提出时，才需要在阶段文档中加入分期或 MVP 视角。

### 6.2 不再把页面结构作为方案设计的起点

原因：

- 直接从页面展开容易把方案做浅。
- 更合理的顺序应该是：先模块，再结构，再流程。

### 6.3 不再把开发阶段理解为编码执行

原因：

- 当前产品定位不是交付代码，而是交付开发可接手的方案。
- 阶段语言必须和真实能力一致，避免误导用户。

## 7. 对现有第一阶段的调整原则

当前第一阶段实现基础可保留，不需要推倒重做。

调整原则是：

1. 保留“通过对话推进需求确认”的机制。
2. 把第一阶段最终文档进一步收紧到以下核心内容：
   - 背景
   - 目标
   - 用户
   - 约束
   - 本次要做的核心内容
   - 待确认问题
3. 文档必须是可确认、可下载的整理文档，而不是零散字段集合。

## 8. 实施建议

建议按以下顺序改造：

1. 先替换前端阶段标题和阶段说明。
2. 再调整后端阶段定义与主流程顺序。
3. 接着修正文案、推荐语和产物命名。
4. 最后分别细化每个阶段的生成规则与确认动作。

## 9. 风险与开放点

当前仍有两个开放点需要后续在实现效果中验证：

1. `交付清单` 阶段是否只做整理下载，还是还需要单独的最终确认文案。
2. `方案设计` 与 `细节确认` 之间的边界在不同项目类型下是否足够稳定。

但这些开放点不影响先按当前版本落一版完整流程，再根据效果调整细节。
