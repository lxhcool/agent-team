# Workspace 前台优先改版设计

日期：2026-05-06  
状态：待评审

## 1. 背景

当前 workspace 流程把内部生产过程直接暴露给用户，核心阶段围绕 `requirements`、`product`、`ui_direction`、`prototype`、`technical`、`development`、`acceptance`、`deployment` 展开。

这套方式对开发者和操作者有用，但不符合普通用户的理解方式。普通用户不关心系统内部怎么拆阶段，更关心两件事：

- 系统有没有理解我的需求
- 有没有持续产出越来越接近目标的结果

因此，这次改版的目标不是给现有阶段换中文名，而是把 workspace 从“开发流程面板”改成“结果优先的创建工作台”。

## 2. 问题定义

当前体验有四个核心问题：

1. 暴露的是内部流程词汇，不是用户能直接理解的进展
2. 用户被迫沿着阶段思考，而不是沿着结果判断
3. 中间产物更像流程检查点，不像持续生成的成果
4. 还没验证前台方向时，就过早进入技术方案和服务端思路

## 3. 产品目标

把一句模糊需求，尽快变成一个可见、可改、可预览的前台结果，并且尽量少打扰用户。

用户应该感受到：

- 系统理解了我的需求
- 系统马上开始做了
- 我一直能看到最新成果
- 我可以随时插话修改
- 更复杂的能力会在前台确认后再补

## 4. 非目标

- 不把 workspace 做成通用开发者控制台
- 不把原始模型、工具、路由错误直接暴露给普通用户
- 不默认先做全栈规划
- 不使用固定的全局阶段结构套所有任务
- 不把“选模板”作为主要交互方式

## 5. 核心原则

1. 结果优先：主界面永远优先展示当前最值得看的成果
2. 少打断：系统少问问题，多做默认补全
3. 阶段动态：用户看到的阶段由需求自动生成，不是固定工程流水线
4. 前台优先：先交付接近可上线的前端页面，再决定是否补后台
5. 可随时打断：用户可以在过程中直接修改方向
6. 复杂能力后置：后台、存储、CMS、权限等在前台稳定后再建议补全

## 6. 主体交互模型

workspace 改成双栏创建工作台：

- 左侧：进展区
- 右侧：成果区

右侧是主舞台，视觉权重应明显高于左侧。建议初始比例：

- 左侧 30% - 35%
- 右侧 65% - 70%

## 7. 左侧：进展区

### 7.1 目标

左侧的作用是建立信任和解释系统正在推进什么。  
它不是聊天记录，不是开发日志，也不是完整推理过程。

### 7.2 结构

每个进展项包含：

- 阶段名
- 当前动作
- 完成状态

### 7.3 规则

- 阶段数量不固定
- 阶段由系统根据用户需求自动生成
- 命名必须使用用户语言
- 阶段结构由系统决定，用户不需要定义
- 左侧只展示整理后的结果，不展示底层执行细节

### 7.4 命名规范

允许的阶段命名风格：

- `明确博客方向`
- `生成核心页面`
- `完善阅读体验`
- `优化展示效果`
- `交付可预览版本`

不允许直接暴露的命名风格：

- `prototype`
- `technical`
- `schema migration`
- `tRPC`
- `requirements analysis`

### 7.5 示例

当用户输入 `帮我做一个个人博客` 时，左侧阶段可以自动生成：

1. 明确博客方向  
   当前动作：已采用默认推荐，按现代极简方向开始
2. 生成核心页面  
   当前动作：正在生成首页、文章列表和文章详情
3. 完善阅读体验  
   当前动作：正在调整排版和内容层级
4. 优化展示效果  
   当前动作：正在优化移动端和页面细节
5. 交付可预览版本  
   当前动作：正在生成完整前端预览

### 7.6 粒度要求

左侧需要让用户看到系统确实在前进，但不能变成长流水账。  
重点是“当前阶段 + 当前动作”，而不是把所有内部事件都堆上去。

## 8. 右侧：成果区

### 8.1 目标

右侧永远展示当前最有价值的成果。  
它不是文件列表，也不是阶段切页，而是持续更新的主预览区。

### 8.2 成果升级顺序

建议按以下顺序自动切换主视图：

1. 推荐方案
2. 首页预览
3. 其他核心页面预览
4. 完整前端预览

### 8.3 规则

- 不要求用户手动推进内部阶段
- 系统自动把最新且最重要的成果顶到主视图
- 保留轻量页面切换入口，便于用户回看已完成页面
- 用户看到的产物名称必须是用户语言

推荐使用的名称：

- `推荐方案`
- `首页预览`
- `页面预览`
- `完整预览`

不建议把以下术语作为默认主标签：

- `设计稿`
- `HTML 原型`
- `阶段产物`

这些词可以存在于内部元数据里，但不应主导普通用户视角。

## 9. 默认启动方式

当用户只给出一句宽泛需求时，采用“推荐 + 立即开工”。

示例：

用户输入：`帮我做一个个人博客`

系统行为：

1. 自动推断一个合理方向
2. 右侧立即展示简短推荐方案
3. 不等待确认，直接开始生成前台第一版
4. 左侧同步更新阶段和当前动作
5. 用户随时可以插话修改

`按默认方案继续` 应该是一种可选回复，而不是强制点击步骤。

## 10. 默认补全规则

为了减少提问但又不至于乱生成，系统必须具备强默认补全能力。  
这不是模板，而是按需求类型进行自动补全。

以个人博客为例，前台第一版默认应尽量补齐：

- 首页
- 关于页
- 文章列表页
- 文章详情页
- 基础导航
- 作者简介区
- 示例内容
- 加载态、空状态、404

第一版的目标不是只生成一张首页，而是生成一个像样的前台产品雏形。

## 11. 修改机制

用户在任何阶段都可以直接打断并修改方向，例如：

- `太素了，改得更有杂志感`
- `去掉关于页`
- `首页加头像和一句简介`

系统收到修改后应当：

- 在左侧追加新的调整动作
- 在现有成果上继续迭代
- 不要求用户重开流程

产品体验应当像协作创作，而不是批处理任务。

## 12. 错误翻译

技术错误不能直接暴露给普通用户。

不要直接显示：

- `500`
- `timeout`
- `tool not found`
- provider/model/tool 原始报错

应该翻译成用户可理解的状态，例如：

- `这一部分生成较慢，先为你展示当前可预览版本`
- `我已切换另一种生成方式，继续补全页面`
- `当前版本已经可以查看，其余页面仍在生成`

原则是：用户看到的是产品状态，不是底层异常。

## 13. 前台完成边界

既然本次策略是前台优先，就必须定义“前台完成”的判断标准。

建议完成条件：

- 核心页面已齐
- 页面间可正常跳转
- 视觉方向基本统一
- 移动端基本可用
- 存在一版完整可预览结果

达到这个边界后，系统不应继续默认扩展，而应进入建议下一步。

## 14. 前台完成后的升级入口

当前台版本被认可后，再由系统建议是否补全：

- 文章发布后台
- 登录与权限
- 图片上传
- 对象存储
- 内容管理

这些都应是“建议项”，不是默认项。

这样可以避免前台方向还没确认，就提前做大量可能白做的服务端设计。

## 15. 对当前 workspace 的改造要求

### 15.1 用户侧改造

应把当前以固定阶段为中心的工作台，改成：

- 左侧动态进展
- 右侧最新成果
- 宽泛需求下立即开始生成
- 基于当前成果持续修订

### 15.2 系统侧改造

系统内部仍然可以保留现有 orchestration 和内部 stage 机制。  
但用户侧不应再直接映射这些内部 key。

更合理的做法是：

- 内部 stage 作为执行元数据
- 用户可见阶段作为展示层，由当前任务和成果进度动态推导

## 16. 可能影响的实现范围

结合当前代码结构，后续实现大概率会集中影响：

- `frontend/src/app/workspaces/[id]/page.tsx`
- `backend/app/api/workspaces.py`
- 当前基于固定 stage 暴露用户流程的推荐和编排逻辑

预期需要新增或调整的能力包括：

- 动态用户阶段展示模型
- 成果优先的主预览选择逻辑
- 前台优先生成默认策略
- 用户安全的错误翻译机制
- 前台完成后的能力升级建议入口

## 17. 本次已确认的产品结论

- 双栏布局：是
- 左侧展示进展，不展示完整推理：是
- 右侧永远优先展示当前成果：是
- 阶段数量固定：否
- 阶段由用户定义：否
- 用户只给一句话时默认立即开始：是
- 默认先做全栈：否
- 第一版应尽量接近可上线前端：是
- 后台与存储作为后续升级建议：是

## 18. 成功标准

如果改版成功，应满足：

1. 非技术用户不需要理解内部阶段也能看懂当前进展
2. 右侧在很短时间内就能出现可见成果
3. 用户可以围绕结果提修改，而不是围绕抽象阶段做决策
4. 前台方向能先被验证，再决定是否进入服务端补全
5. 内部失败不再以原始技术报错形式泄漏到普通用户界面

## 19. 落地提醒

这次改版必须是交互模型和状态呈现方式的改造，而不是简单给现有固定阶段换一层文案。

如果只是把现有阶段重新命名，但用户仍然被迫跟着固定流程走、仍然看不到持续产出的前台成果，那么核心问题不会解决。
