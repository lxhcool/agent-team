# Stage Expert Group Design

日期：2026-05-14

## 1. 背景

当前产品主线已经从独立 Agent 协作转向工作区流程：用户输入需求，系统按阶段生成可确认、可交接的产物。

现有流程的基础效果可用，但仍有一个关键差异化不足：

用户看到的主要还是“一个助手给出一份阶段方案”。如果用户本身是小白，就很难判断方案是否合理、是否遗漏关键问题、是否可以继续进入下一阶段。

因此需要在现有流程模式中加入专业多 Agent 机制。这个机制不再沿用圆桌模式，也不做独立聊天页面，而是作为每个阶段的后台专家组工作流。

## 2. 产品定位

阶段专家组是流程阶段内的专业审查与收束机制。

它的目标不是制造更多对话，而是让每个阶段产物经过不同专业视角的独立检查，并把检查结果整理成用户可以理解、可以确认、可以追踪的结论。

核心差异：

- 普通 AI 网页：一个模型直接写一份方案。
- Team Agent 流程：阶段主 Agent 生成方案，多个专家 Agent 独立审查，收束 Agent 合并意见，最终形成可确认产物和决策记录。

## 3. 非目标

本次设计明确不做：

1. 不恢复或迁移圆桌模式。
2. 不展示多 Agent 实时聊天气泡。
3. 不把专家意见直接自动改写进正文。
4. 不改坏现有阶段主生成逻辑。
5. 不引入外部搜索作为第一版必需能力。
6. 不做专家市场、可视化会议、复杂编排器。

## 4. 设计原则

### 4.1 保持现有流程稳定

第一版不替换现有阶段生成逻辑。

现有阶段助手继续负责生成阶段草案。专家组只在草案生成后做审查、解释和建议。

### 4.2 多 Agent 必须真实分工

多 Agent 不是同一段 prompt 换几个称呼。

每个专家 Agent 必须有明确职责、输入、输出格式和判断标准，并独立给出结论。

### 4.3 用户只看收束结果

用户不需要阅读所有专家的长篇发言。

前台默认展示收束后的专业检查报告，同时允许展开查看各专家原始意见。

### 4.4 专业结论必须可追踪

每次专家组检查都应留下记录：

- 本次检查基于哪个阶段草案
- 哪些专家参与
- 发现了哪些风险
- 哪些建议被采纳
- 哪些建议暂不采纳
- 为什么建议继续或暂停进入下一阶段

## 5. 第一版范围

第一版只接入三个关键阶段：

1. 方案设计
2. 细节确认
3. 开发方案

暂不接入：

- 需求确认：避免过早增加复杂度。
- 最终交付：第一版先不做全局一致性检查。

## 6. 阶段专家组配置

### 6.1 方案设计阶段

目标：检查产品方案是否完整、清晰、可继续细化。

专家角色：

- 产品结构专家：检查功能模块、页面结构、主流程是否完整。
- 用户体验专家：检查用户路径是否顺、关键操作是否清楚。
- 技术可行性专家：检查方案是否过重、是否存在明显落地风险。

质量标准：

- 是否有明确入口、主操作和结果反馈。
- 是否覆盖核心用户路径。
- 是否说明管理侧或运营侧。
- 是否有关键异常流程。
- 是否存在明显过度设计。
- 是否足以进入细节确认。

### 6.2 细节确认阶段

目标：锁住最容易造成返工的业务规则和边界。

专家角色：

- 业务规则专家：检查规则是否清楚、是否有互相冲突。
- 状态权限专家：检查角色、权限、状态流转是否完整。
- 返工风险专家：识别后续开发中最容易返工的空白点。

质量标准：

- 是否定义关键角色与权限。
- 是否定义核心状态流转。
- 是否覆盖异常处理。
- 是否说明数据口径。
- 是否列出未决问题及影响。
- 是否足以进入开发方案。

### 6.3 开发方案阶段

目标：检查开发方案是否能被真实开发者接手。

专家角色：

- 技术架构专家：检查模块边界、技术路线和依赖。
- 数据/API 专家：检查数据结构、接口组织和关键约束。
- QA 验收专家：检查验收标准、测试风险和交付判断。

质量标准：

- 是否有清晰模块拆分。
- 是否说明数据与接口组织。
- 是否有实施顺序。
- 是否暴露关键风险。
- 是否有验收口径。
- 是否足以交给开发接手。

## 7. 专家输出格式

每个专家 Agent 输出固定结构：

```text
结论：通过 / 有风险 / 不建议进入下一阶段
发现的问题：
- ...

建议修改：
- ...

是否阻塞：
- 是 / 否

需要用户确认：
- ...
```

要求：

1. 每个专家最多输出 3 个主要问题。
2. 必须区分阻塞问题和普通优化。
3. 不允许泛泛而谈。
4. 必须基于当前阶段草案和已确认上下文。

## 8. 收束输出格式

收束 Agent 汇总专家意见后，给用户展示一份专业检查报告：

```text
总体判断：
当前方案基本成立 / 需要补充后再继续 / 不建议进入下一阶段

为什么这样判断：
- ...

主要风险：
- ...

专家分歧：
- ...

建议补充：
- ...

你需要重点看：
- ...

是否建议进入下一阶段：
是 / 否 / 需要先确认以下问题

需要你确认的问题：
- ...
```

## 9. 冲突识别

专家组必须允许出现冲突。

典型冲突：

- 产品结构专家建议增加功能，技术可行性专家认为会明显增加首版复杂度。
- 用户体验专家建议简化流程，业务规则专家认为会丢失必要状态。
- QA 专家认为验收口径不足，技术架构专家认为当前方案已经可开发。

收束 Agent 不应抹平冲突，而应把冲突转成用户可理解的取舍：

```text
存在一个关键取舍：
产品视角认为会员等级能提升运营能力；
技术视角认为首版加入会员等级会增加权限、权益和价格规则复杂度。

建议：
首版只保留用户标签，不做完整会员体系。
```

## 10. 决策记录

专家组检查后，如果形成明确取舍，应记录为阶段决策。

决策记录格式：

```text
决策：
首版不做会员等级。

原因：
当前核心目标是完成预约闭环，会员体系会引入权益、价格、权限和运营规则。

影响：
后续可以在用户标签和订单记录基础上扩展。
```

第一版可以先把决策记录存入检查结果中，不单独做全局决策中心。

## 11. 用户交互

在阶段页面中加入模块：

```text
专业视角检查
```

状态：

- 未检查：显示“生成专业检查”按钮。
- 检查中：显示进度和参与专家。
- 已完成：显示收束报告。
- 失败：显示失败原因，允许重试。

第一版动作：

- 生成专业检查
- 展开专家原始意见
- 重新生成检查

暂不做：

- 自动采纳建议
- 自动重写阶段正文
- 多轮专家追问

## 12. 后端概念

建议新增流程内概念：

```text
workspace_stage_reviews
```

核心字段：

- id
- workspace_id
- stage_id
- stage_key
- status
- review_type
- draft_message_id
- participants_json
- expert_findings_json
- summary
- result_json
- created_at
- updated_at

API 建议：

```text
POST /workspaces/{workspace_id}/stages/{stage_key}/review
GET  /workspaces/{workspace_id}/stages/{stage_key}/reviews
GET  /workspaces/{workspace_id}/stages/{stage_key}/reviews/{review_id}
```

## 13. Skill 与 Agent 的关系

Skill 继续作为专业规则和输出约束。

Agent 重新定位为后台专家角色，不再作为前台聊天角色。

建议新增 Skill：

- stage-review-product-risk
- stage-review-user-clarity
- stage-review-technical-feasibility
- stage-review-rule-coverage
- stage-review-qa-readiness
- stage-review-synthesis

建议新增或重命名 Agent：

- product-structure-reviewer
- ux-clarity-reviewer
- technical-feasibility-reviewer
- rule-boundary-reviewer
- qa-readiness-reviewer
- stage-review-synthesizer

## 14. 实施顺序

第一步：只做方案设计阶段试点。

流程：

1. 用户完成方案设计阶段草案。
2. 点击生成专业检查。
3. 后端基于当前阶段草案和前序确认内容生成专家意见。
4. 收束 Agent 输出检查报告。
5. 前端展示检查报告。

第二步：扩展到细节确认和开发方案。

第三步：加入“按建议修改方案”，但必须由用户显式触发。

第四步：清理旧圆桌相关前台和后台遗留。

## 15. 成功标准

第一版成功标准：

1. 用户能清楚知道当前阶段方案是否基本可靠。
2. 用户能看到专业角色发现的关键风险。
3. 用户能理解哪些问题必须自己确认。
4. 现有阶段生成效果不被破坏。
5. 专家组结果能明显区别于普通 AI 网页的一次性方案输出。
