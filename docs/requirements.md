# Team Agent 需求文档

## 1. 文档目标

本文档定义 Team Agent 的新产品边界，回答四个问题：

1. 这个系统现在解决什么问题。
2. 它服务谁。
3. 它的主流程和阶段产物是什么。
4. 它明确不做什么。

## 2. 产品定位

Team Agent 是一个 `面向小团队的 AI 交付经理工作台`。

它不负责代码生成，不负责本地执行开发任务，不负责替代 IDE。

它负责把：

- 新项目的一句话需求
- 现有项目的迭代诉求
- 已有 PRD、会议纪要、聊天记录、设计说明等零散材料

推进成 `可评审、可确认、可开工` 的阶段化交付成果。

## 3. 目标用户

当前主要服务：

- 小团队项目负责人
- 创始人或业务 owner
- 兼任产品经理的负责人

这类用户通常需要协调设计、开发和其他协作者，但缺少稳定的前期交付收敛流程。

## 4. 核心问题

Team Agent 重点解决以下问题：

1. 原始需求模糊，无法直接开工。
2. 讨论分散，信息散落在多个地方。
3. 方案不成体系，难以交接给设计或开发。
4. 每个阶段缺少明确确认动作，导致反复返工。
5. 迭代需求进入时，无法快速识别影响范围和待补充信息。

## 5. 主流程

系统支持两类入口：

### 5.1 新项目开始

用户输入一句自然语言需求，系统从零开始澄清、收敛和推进。

### 5.2 现有项目迭代

用户上传已有材料，系统先理解现状，再识别这次迭代的变更和影响。

### 5.3 统一主线

无论入口如何，都会进入同一流程：

1. 理解输入
2. 识别缺口
3. 发起澄清
4. 生成阶段产物
5. 评审与修订
6. 阶段确认
7. 汇总当前交付成果

## 6. 阶段设计

每个阶段都必须有自己的目标、输入、产物、确认动作和修订记录。

### 6.1 需求澄清

目标：
- 把原始描述补成清晰问题定义

产物示例：
- 背景
- 目标
- 非目标
- 关键角色
- 当前约束
- 待澄清问题

### 6.2 范围定义

目标：
- 明确本次到底做什么、不做什么

产物示例：
- 功能范围
- 优先级
- 页面或模块范围
- 本期边界
- 依赖事项

### 6.3 方案设计

目标：
- 形成可讨论、可评审的方案表达

产物示例：
- 用户流程
- 信息架构
- 页面或模块关系
- 关键交互说明
- 特殊业务规则

### 6.4 实现准备

目标：
- 让设计、开发或协作者可以接手

产物示例：
- 开发说明
- 模块拆分建议
- 接口或数据约束说明
- 风险项
- 前置条件
- 验收标准

### 6.5 交付总览

目标：
- 汇总所有阶段结果，而不是重新生成一份大文档

产物示例：
- 当前版本说明
- 各阶段产物索引
- 已确认事项
- 待确认事项
- 建议下一步

## 7. 产物原则

系统不是只在最后导出一个文件，而是 `每个阶段先沉淀产物，最后再统一索引`。

每个产物建议具备稳定骨架：

- 标题
- 所属阶段
- 当前版本
- 结论摘要
- 关键决定
- 待确认项
- 修订记录

但具体内容组织不固定，应由 AI 根据项目类型动态编排。

## 8. 核心原则

1. 不做代码生成
2. 不做 CLI 执行
3. 不做自动改仓库
4. 不做远程控制 IDE
5. 每一阶段必须有可确认产物
6. 产物采用弱模板、强结构、动态编排

## 9. 非目标

当前明确不做：

- 本地代码执行器
- 自动测试与构建
- 执行结果回传链路
- 在线完成完整研发闭环
- 通用项目管理平台
- 通用文档平台

## 10. 成功标准

这个产品是否成立，不看“有没有写代码”，看：

1. 用户是否更快进入可开工状态
2. 阶段产物是否足够支撑团队协作
3. 需求变化是否被清晰记录和收束
4. 开发前的理解偏差和返工是否明显减少
