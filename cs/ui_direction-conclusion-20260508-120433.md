## 已确认内容

### 1. 角色与权限
*   **后台管理员**：系统仅设置一个**超级管理员**账号，拥有后台所有功能的完整操作权限，包括订单审核、状态更新及服务信息管理。无需设计复杂的权限体系。

### 2. 预约单状态流转
已明确预约单从创建到结束的完整状态链及触发动作：
*   **状态定义**：
    *   **待审核**：用户提交预约后的初始状态。
    *   **处理中**：管理员审核通过后的状态。
    *   **已完成**：管理员手动标记服务完成后的最终状态。
    *   **已拒绝**：管理员审核拒绝后的最终状态。
*   **流转规则**：
    1.  用户提交预约 -> 系统生成订单，状态为 **“待审核”**。
    2.  管理员操作 **“通过”** -> 状态变更为 **“处理中”**。
    3.  管理员操作 **“拒绝”** -> 状态变更为 **“已拒绝”**（流程结束）。
    4.  管理员在订单为“处理中”时，手动操作 **“标记完成”** -> 状态变更为 **“已完成”**（流程结束）。

### 3. 预约时段规则
*   **时段划分**：采用**按小时划分**的固定时段。
*   **可选时段**：从 **09:00-10:00** 开始，至 **20:00-21:00** 结束，共计12个可预约时段。
*   **容量限制**：每个时段的**最大预约人数为5人**。当某时段预约人数达到上限时，该时段对用户不可选。

### 4. 设计约束
继续遵循前期确立的核心规则：
*   **无支付流程**：预约流程不涉及任何在线支付。
*   **单店模式**：后台仅服务于单一门店。
